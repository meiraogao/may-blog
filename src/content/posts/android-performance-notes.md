## 为什么从体验问题开始

性能优化最容易陷入指标崇拜。启动耗时、帧率、内存峰值、包体积都很重要，但真正推动工程决策的，应该是用户能感受到的问题。

比如：

- 首页首屏是否足够快
- 列表滑动是否稳定
- 后台回来是否容易白屏
- 低端设备是否出现明显掉帧

指标是地图，体验才是目的地。

## 一套粗粒度排查顺序

我通常会先把问题拆成四层：

1. 启动路径
2. 页面渲染
3. 数据加载
4. 后台资源

这种拆法不一定精细，但适合在复杂项目里快速定位方向。

```kotlin
class StartupTracker {
    fun mark(name: String) {
        // 接入真实埋点前，可以先在本地统一记录关键节点
        println("startup marker: $name")
    }
}
```

## 优化要能回归

一次优化如果不能被回归验证，很快就会变成口口相传的经验。

我更倾向于建立轻量的性能检查清单，把关键路径变成团队能持续执行的东西。
